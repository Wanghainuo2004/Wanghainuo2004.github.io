一、Agent Loop 的基本运转方式

▼
核心概念

Agent 的本质：一个永不疲倦的循环

Agent 不是一个智能体，而是一个循环。它的核心是一个 while True：感知环境 → 思考决策 → 执行动作 → 获取反馈 → 再次感知。所有复杂的 Agent 系统，拆到底层都是这个简单循环的不同变体。

感知 (Perceive) → 决策 (Decide) → 行动 (Act) → 反馈 (Observe) ↺
代码实现

不到 20 行的核心代码

一个最简 Agent Loop 的 Python 实现，展现了感知-决策-行动-反馈的完整循环：

def agent_loop(model, tools, task): messages = [{"role": "user", "content": task}] while True: # 决策：让模型思考下一步 response = model.chat(messages, tools=tools) messages.append(response) # 如果模型认为任务完成，退出循环 if response.finish_reason == "stop": return response.content # 行动：执行工具调用 for call in response.tool_calls: result = execute_tool(call) messages.append(tool_result(call, result))
关键洞察

Agent Loop 的复杂度不在循环本身，而在循环的终止条件和错误处理。一个没有退出条件的 Agent 是一个无限循环炸弹。

运行机制

三个关键设计决策

  • 何时停止：模型输出 finish_reason == "stop"，或者达到最大轮次
  • 何时重试：工具调用失败时的重试策略与退避机制
  • 何时放弃：连续失败 N 次后优雅降级，而非死循环

知识检查：Agent Loop 的核心本质是什么？

一个复杂的神经网络模型
一个感知-决策-行动-反馈的循环结构
一个调用 API 的函数
一个数据库查询引擎

二、Workflow 和 Agent 的区别

▼
核心区分

控制权在谁手里？

Workflow 和 Agent 的本质区别不在于是否有 LLM 参与，而在于控制权归属：Workflow 的控制权在开发者手里（预定义路径），Agent 的控制权在模型手里（动态决策）。

维度 Workflow Agent
控制流 开发者预定义 模型动态决策
确定性 高，可预测 低，有随机性
灵活性 受限于预定义路径 理论上无限
调试难度 低，路径固定 高，需要 Trace
适用场景 流程明确的重复任务 开放性、探索性任务
五种控制模式

Anthropic 总结的五种 Workflow 模式

  • Prompt Chaining（提示链）：任务分解为顺序步骤，前一步输出是后一步输入
  • Routing（路由）：根据输入分类，分发到不同的专用处理流程
  • Parallelization（并行）：多个子任务同时执行，汇总结果
  • Orchestrator-Worker（编排器-工作者）：一个编排器动态分配任务给多个工作者
  • Evaluator-Optimizer（评估器-优化器）：生成-评估-优化的迭代循环
Prompt Chaining → Routing → Parallelization → Orchestrator-Worker → Evaluator-Optimizer
实践建议

能用 Workflow 解决的问题，不要用 Agent。Agent 的灵活性是以可控性为代价的。先尝试最简单的 Prompt Chaining，不够再逐步升级到更复杂的模式。

知识检查：Workflow 和 Agent 最本质的区别是什么？

Workflow 不用 LLM，Agent 用 LLM
Workflow 更快，Agent 更慢
控制权在开发者手里（Workflow）还是在模型手里（Agent）
Workflow 只能处理文本，Agent 能调用工具

三、为什么 Harness 比模型更关键

▼
核心论点

模型是引擎，Harness 是车身

Harness（驾驭层）是包裹在模型外面的工程系统：它决定模型能做什么、不能做什么、做错了怎么办。OpenAI 的实践表明：他们的 Agent 系统有100万行代码，其中模型调用只占极小一部分，绝大多数是 Harness 工程。

四大支柱

Harness 的四个核心职责

🎯

验收基线

定义"什么算完成"：输出格式校验、业务规则检查、质量门禁

🚫

执行边界

定义"能做什么"：权限控制、操作白名单、资源配额限制

📣

反馈信号

定义"做得怎样"：错误信息、进度报告、质量评分

🔄

回退手段

定义"做错了怎么办"：重试、降级、人工介入、事务回滚

用户请求 ↓ Harness：校验 & 边界检查 ↓ LLM 推理 ↓ Harness：输出校验 & 质量门禁 ↓ 工具执行 & 结果返回 ↓ Harness：结果验证 & 反馈组装
OpenAI 的教训

百万行代码中，模型调用只是一行 API 请求。其余 99.9% 是 Harness：输入清洗、输出校验、错误重试、权限控制、日志追踪、降级策略。模型能力越强，Harness 越重要——因为模型犯错的代价更大了。

反直觉的认知

模型升级 ≠ 系统升级

很多人以为换个更强的模型就能解决问题。实际上，没有好的 Harness，更强的模型只会更高效地犯错。Harness 是 Agent 系统的"安全气囊"——平时看不见，出事时救命。

四、上下文工程为什么决定稳定性

▼
核心概念

Context Rot：上下文腐烂

当上下文窗口塞入过多信息时，模型的注意力会被稀释，导致"看到了但没理解"——这就是 Context Rot。不是窗口不够大，而是信息太多模型处理不过来。上下文工程的本质是：在有限窗口内放入最有价值的信息。

分层管理

上下文的三层架构

  • 系统层（System）：角色定义、行为约束、工具说明——长期不变
  • 会话层（Session）：对话历史、中间状态——随会话变化
  • 任务层（Task）：当前任务的具体信息——每次调用可能不同
三种压缩策略

如何管理有限的上下文窗口

截断 (Truncation) | 摘要 (Summarization) | 选择 (Selection)
策略 原理 优点 缺点
截断 丢弃最早的消息 简单高效 可能丢失关键信息
摘要 用 LLM 压缩历史 保留核心信息 增加延迟和成本
选择 按相关性检索历史 精准保留上下文 实现复杂度高
成本优化

Prompt Caching 与 Skills 延迟加载

  • Prompt Caching：将不变的系统提示缓存，避免重复计算。可节省 50-90% 的 Token 成本
  • Skills 延迟加载：不把所有工具说明塞进系统提示，而是按需动态注入。只有当任务需要某个能力时，才加载对应的工具描述
工程实践

上下文不是越大越好。一个精心组织的 4K 上下文，效果往往好于一个杂乱无章的 128K 上下文。质量 > 数量。

知识检查：什么是 Context Rot？

上下文信息过多导致模型注意力稀释，理解能力下降
上下文窗口物理损坏导致数据丢失
模型忘记了之前的对话内容
Token 数量超过限制导致报错

五、工具设计决定 Agent 能做什么

▼
演进历程

三代工具设计范式

工具设计经历了从"直接封装"到"为模型设计"的演进，每一代都在解决上一代的核心痛点。

第一代：API 封装 → 第二代：ACI → 第三代：Advanced Tool Use
ACI 设计原则

Agent-Computer Interface：为模型设计的接口

ACI（Agent-Computer Interface）的核心思想是：工具接口不应该按人类的习惯设计，而应该按模型的理解方式设计。关键原则包括：

  • 语义清晰：工具名和参数名要自解释，让模型一看就懂
  • 粒度适中：太粗的工具模型不知道怎么用，太细的工具需要太多调用步骤
  • 错误友好：返回的错误信息要能帮模型理解问题并自我修正
  • 幂等安全：相同调用多次执行的结果一致，避免模型重试时产生副作用
  • 最小权限：每个工具只暴露完成任务所需的最少能力
代际 设计思路 典型问题
第一代 API 封装 直接把 REST API 包成工具 参数太多、命名不直观、模型容易出错
第二代 ACI 为模型重新设计接口 静态工具集，无法应对动态场景
第三代 Advanced 工具可组合、可发现、可学习 复杂度高，需要更多工程投入
Anthropic 的建议

花在工具设计上的时间应该和花在 prompt engineering 上的时间一样多。一个设计良好的工具，比十段精心编写的提示词更有效。

常见错误

工具设计的三个陷阱

  • 工具太多：给模型 50 个工具，它选择的准确率会大幅下降。精选 10-15 个核心工具
  • 描述太差：工具描述写得像 API 文档而不是给人看的说明。要写成"这个工具做什么，什么时候用"
  • 没有示例：在工具描述中加入输入输出示例，能显著提升模型调用准确率

六、记忆系统如何设计

▼
核心框架

四种记忆类型

Agent 的记忆系统借鉴了认知科学的记忆分类，分为四种类型，每种有不同的存储和检索机制。

💡

工作记忆

当前任务的上下文信息，相当于上下文窗口。容量有限，生命周期短

💪

程序性记忆

"怎么做"的技能和流程。对应系统提示中的行为规则和工具使用规范

📄

情景记忆

过去经历的具体事件。存储对话历史、任务执行记录，可按时间检索

📚

语义记忆

抽象的知识和事实。存储在向量数据库或知识图谱中，可按语义检索

用户输入 ↓ 工作记忆：加载相关上下文 ↓ 语义记忆：检索相关知识 ↓ 程序性记忆：遵循行为规则 ↓ LLM 生成响应 ↓ 情景记忆：记录本次经历
整合机制

记忆不是存储，而是检索

记忆系统的难点不在存储，而在何时检索什么。一个好的记忆系统需要：

  • 相关性检索：基于当前任务语义，检索最相关的历史信息
  • 时效性衰减：越久远的记忆权重越低，除非被频繁引用
  • 冲突消解：当新旧记忆矛盾时，以最新且可信度高的为准
  • 主动遗忘：定期清理过时或低价值的记忆，避免噪音积累
工程现实

大多数生产级 Agent 的记忆系统还停留在"对话历史 + RAG"阶段。真正实现四种记忆无缝整合的系统极少，这是一个开放的工程挑战。

七、如何逐步放开 Agent 自主度

▼
核心挑战

长任务需要跨 Session 存活

当 Agent 的任务从"回答一个问题"演变为"完成一个项目"时，单个 Session 不够用了。任务可能持续数小时甚至数天，中间可能因为各种原因中断。这就需要：任务状态外化。

三大机制

支撑长任务的技术基础

💾

任务状态外化

将任务进度、中间结果、决策记录持久化到外部存储，而非仅保存在内存中

🔌

后台 I/O

工具调用不阻塞主循环：发邮件、部署代码、等待审批等异步操作在后台执行

🔃

断点续传

Session 中断后能从上次断点恢复执行，而非从头开始

小任务：单 Session → 中任务：状态外化 → 大任务：后台 I/O + 断点续传 → 持续任务：定时调度
自主度阶梯

不是一步到位，而是逐步放开

  • Level 0：人在环中，每步确认（审批模式）
  • Level 1：人设目标，Agent 执行，异常时暂停
  • Level 2：Agent 自主执行，事后汇报
  • Level 3：Agent 持续运行，定期同步（cron/heartbeat 模式）
风险提示

自主度每提升一级，风险指数级增长。Level 2 以上的 Agent 需要完善的监控、告警和回退机制。不要跳级——先在低风险场景验证，再逐步扩大权限。

八、多 Agent 如何组织

▼
组织模式

指挥者 vs 统筹者

多 Agent 系统有两种基本组织模式，选择哪种取决于任务的复杂度和可分解性。

模式 指挥者 (Commander) 统筹者 (Orchestrator)
控制方式 直接分配任务，统一调度 制定计划，让子 Agent 自主执行
适用场景 任务可明确分解 任务需要探索和迭代
通信方式 星型拓扑，所有通信经过中心 网状拓扑，Agent 间可直接通信
故障影响 中心挂了全挂 单点故障影响有限
子 Agent 设计

每个子 Agent 是一个独立的 Agent Loop

子 Agent 不是简单的函数调用，而是拥有自己的 Agent Loop、工具集和上下文的独立实体。它们通过协作协议进行通信，而非共享内存。

  • 消息传递：通过结构化消息交换信息（类似 Actor 模型）
  • 任务队列：通过任务队列异步协作，解耦生产者和消费者
  • 共享黑板：所有 Agent 读写同一个"黑板"，通过状态变化触发协作
幻觉放大效应

多 Agent 系统的最大风险

单个 Agent 的幻觉率是 5%。三个 Agent 串联，每个的输出是下一个的输入，整体幻觉率会指数级放大。这不是简单的 5% x 3，而是误差的累积传播。

Agent A 输出（5% 幻觉率） ↓ Agent B 处理（基于 A 的输出，10% 幻觉率） ↓ Agent C 处理（基于 B 的输出，15%+ 幻觉率）
对抗幻觉放大

关键策略：每个 Agent 的输出必须经过独立验证后才传递给下游。不要让下游 Agent 盲信上游的输出。加入校验 Agent（Evaluator）作为"质检员"。

知识检查：多 Agent 系统中"幻觉放大"的原因是什么？

Agent 之间通信延迟导致信息丢失
多个 Agent 同时产生相同的幻觉
上游 Agent 的幻觉作为输入传递给下游，误差累积传播
模型参数过多导致过拟合

九、Agent 评测如何做

▼
核心指标

Pass@k vs Pass^k：两种不同的成功定义

  • Pass@k：运行 k 次，至少有一次成功的概率。衡量的是"能不能做对"
  • Pass^k：运行 k 次，每次都成功的概率。衡量的是"稳不稳定"

Pass@k = 90% 看起来很好，但如果 Pass^k 只有 35%，意味着这个 Agent 10 次里有 6 次会出错。

Pass@5 = 95% vs Pass^5 = 60% → Agent 不够稳定
三类评分器

如何判断 Agent 的输出好不好

评分器类型 原理 适用场景
规则评分器 基于明确规则判定（格式、正则、断言） 输出格式确定的任务
模型评分器 用另一个 LLM 评估输出质量 开放式、需要语义理解的任务
人工评分器 人类专家打分 高风险、需要领域判断的场景
方法论

先修评测，再改 Agent

很多人的做法是：先改 Agent 代码 → 看看效果好不好 → 再补测试。正确的做法是反过来：

  • 先建立评测基准（baseline）
  • 收集一批有代表性的测试用例
  • 跑出当前的通过率
  • 每次改动都重新评测，用数据说话
最佳实践

评测集要包含三类用例：正常路径（happy path）、边界情况（edge case）、对抗样本（adversarial）。只测正常路径的评测集是自欺欺人。

知识检查：Pass@k 和 Pass^k 的区别是什么？

Pass@k 是通过率，Pass^k 是错误率
Pass@k 是"至少成功一次"的概率，Pass^k 是"每次都成功"的概率
Pass@k 是单次测试，Pass^k 是多次测试
两者是同一个指标的不同写法

十、如何追踪 Agent 执行过程

▼
核心概念

Trace：Agent 的黑匣子

Agent 的决策过程是不透明的——你不知道它为什么选择这个工具、为什么得出这个结论。Trace（追踪记录）就是 Agent 的"黑匣子飞行记录器"，记录每一步的输入、输出、决策和耗时。

两层可观测性

宏观监控 + 微观调试

  • 宏观层：整体成功率、平均延迟、Token 消耗、工具调用分布——用于监控和告警
  • 微观层：单次执行的完整 Trace——用于调试和优化。包含每轮循环的 messages、tool_calls、latency、token_usage
用户请求 ↓ Trace 开始（span_id, timestamp） ↓ Round 1: LLM 调用 + 工具执行 ↓ Round 2: LLM 调用 + 工具执行 ↓ Round N: 最终响应 ↓ Trace 结束（total_tokens, total_latency）
事件流底座

基于事件流的统一追踪架构

所有可观测性都建立在一个事件流（Event Stream）之上。Agent 执行过程中的每个关键节点都会发出一个事件：

  • agent.start — Agent 开始执行
  • llm.call — 模型调用（输入 messages、输出 response、latency）
  • tool.call — 工具调用（工具名、参数、结果、耗时）
  • agent.think — 模型的推理过程
  • agent.end — Agent 执行结束（最终结果、总耗时）
  • agent.error — 异常和错误
工具选型

主流的 Agent 可观测性工具：LangSmith（LangChain 生态）、Braintrust、Arize Phoenix、Helicone。选择标准：是否支持你用的框架、是否有好的 Trace 可视化、是否支持自定义事件。

十一、用 OpenClaw 看 Agent 如何落地

▼
架构设计

五层解耦架构

OpenClaw 是一个开源的 Agent 框架，它的核心设计思想是五层解耦——每一层只关注自己的职责，层与层之间通过接口通信。

L1: 接入层（API Gateway、认证、限流） ↓ L2: 协调层（Agent Loop、任务调度） ↓ L3: 能力层（工具、模型、记忆） ↓ L4: 数据层（状态持久化、Trace 存储） ↓ L5: 基础设施层（消息队列、缓存、监控）
系统提示分层

把 Prompt 当代码管理

OpenClaw 的系统提示不是一坨文本，而是分层管理的模块：

  • 身份层：角色定义和核心约束——几乎不变
  • 能力层：工具说明和使用规范——随能力变化
  • 任务层：当前任务的具体指令——每次不同
  • 安全层：红线规则和合规约束——绝对不可绕过
调度机制

Cron 与 Heartbeat

  • Cron：定时触发 Agent 执行周期性任务（如每日报告、数据同步）
  • Heartbeat：Agent 定期发送心跳信号，报告自身状态。如果心跳中断，系统自动触发恢复流程
安全边界

Agent 的权限不是越少越好，而是刚刚好

OpenClaw 的安全设计遵循最小权限原则，但不是简单的"什么都禁止"：

  • 沙箱执行：代码执行在隔离环境中，无法访问主机文件系统
  • 操作审批：高风险操作（删除数据、发送邮件）需要人工确认
  • 资源配额：Token 消耗、API 调用次数、执行时间都有上限
  • 审计日志：所有操作都有完整记录，可追溯可审计
落地启示

OpenClaw 的价值不在于它有多先进，而在于它展示了一个生产级 Agent 系统的真实面貌：大量枯燥但必要的工程细节。Agent 落地，80% 是工程，20% 是 AI。

十二、常见反模式

▼
反模式 1

全能 Agent 妄想

症状：试图用一个 Agent 解决所有问题，给它所有工具和所有权限。

后果：上下文爆炸、工具选择混乱、安全风险剧增。

修复：拆分为多个专用 Agent，每个 Agent 只负责一个领域。用 Orchestrator 模式协调。

反模式 2

Prompt 堆砌

症状：遇到问题就往系统提示里加规则，Prompt 越来越长。

后果：Context Rot，模型注意力稀释，规则之间相互矛盾。

修复：将规则结构化、分层化。用工具实现约束，而不是用文字描述约束。

反模式 3

无 Trace 盲飞

症状：只看最终结果，不记录中间过程。出了问题无法复现。

后果：调试效率极低，同类问题反复出现。

修复：从第一天就接入 Trace 系统。每个 Agent Loop 的每一轮都要记录。

反模式 4

跳过评测直接优化

症状：凭感觉改 Prompt、换模型，没有量化指标衡量效果。

后果：改了 A 好了但 B 变差，不知道到底有没有进步。

修复：先建评测集，跑出 baseline，每次改动都对比数据。

反模式 5

过度自主

症状：一开始就给 Agent 最高自主权，让它"自己决定一切"。

后果：Agent 在你不了解的场景做出高风险操作，造成不可逆损失。

修复：从 Level 0（人在环中）开始，验证稳定后再逐步放开。每个 Level 都要有对应的监控和回退机制。

反模式 6

工具泛滥

症状：给 Agent 接入几十个工具，觉得"能力越强越好"。

后果：模型选择工具的准确率大幅下降，调用链路变长，调试困难。

修复：精选核心工具（10-15个），提供清晰的工具描述和示例。用"工具路由器"按需加载。

反模式的共同特征

所有反模式的根源都是同一个：低估了工程复杂度，高估了模型能力。Agent 不是"接个 API 就完事"，而是一个需要精心设计的软件系统。

知识检查：以下哪个不是 Agent 开发的常见反模式？

全能 Agent 妄想——用一个 Agent 解决所有问题
Prompt 堆砌——遇到问题就往系统提示加规则
跳过评测直接优化——凭感觉改代码
渐进式放开自主度——从低权限开始逐步提升

Workflow vs Agent

Harness > Model

记忆系统

从 Level 0 开始逐步放开，不要跳级

多 Agent 协作

评测先行
