反面教材：第一代的五个痛点

第一代上下文压缩采用"撑不住才动手"的被动策略，在实际使用中暴露出一系列严重问题。

痛点一：压缩滞后

等到上下文快满才触发压缩，此时已无足够空间执行压缩操作本身（压缩摘要也需要 token），导致"要压缩时已经压不动了"。

痛点二：信息断裂

粗暴截断或简单摘要丢失关键上下文，Agent 忘记之前的决策和承诺，导致行为前后矛盾、重复犯错。

痛点三：缓存失效

滑动窗口截断导致前缀变化，Anthropic / OpenAI 的 prompt cache 全部失效，每次调用都是冷启动，成本飙升。

痛点四：用户无感知

压缩在后台静默发生，用户不知道哪些信息被丢弃了，直到 Agent 犯错才发现"它忘了"。

痛点五：压缩质量不可控

用 LLM 做摘要本身就可能出错——遗漏关键信息、曲解语义、丢失精确数值。压缩质量取决于模型能力，但多数系统对压缩质量没有任何校验机制。

第二代各家做法详解

Claude Code 五段流水线

Claude Code 的核心设计思想是本地操作优先——能用确定性操作解决的，绝不动用 LLM。五级渐进式压缩，从轻到重：

Stage 1 Budget
Reduction
→
Stage 2 Snip
→
Stage 3 Microcompact
→
Stage 4 Context
Collapse
→
Stage 5 Auto-
Compact

当 token 使用接近预算阈值时，系统发出软限制预警。这是一种"提前打招呼"的机制，让后续压缩操作有充足的缓冲空间。

  • 不丢弃任何信息，仅发出警告
  • 为后续阶段争取操作空间
  • 用户可感知当前上下文健康度

纯本地操作，不调用 LLM。根据规则裁剪低价值内容：

  • 截断过长的工具输出（如大文件内容、搜索结果列表）
  • 移除已成功执行的工具调用的冗余返回值
  • 保留工具调用的结构和关键摘要行
  • 零成本、零延迟、零风险

对高频出现的短对话片段进行局部压缩，保持对话流的完整性：

  • 合并连续的简单工具调用为摘要
  • 压缩重复的确认/反馈循环
  • 保留意图和结论，丢弃中间过程

对较长的对话段落进行结构化折叠，使用 LLM 生成高质量摘要：

  • 识别对话中的"任务边界"，按任务段落折叠
  • 保留每个任务的输入、输出、关键决策
  • 摘要中包含结构化标签便于检索
  • 仅在 Stage 2/3 不足以解决问题时触发

最终兜底方案。当所有轻量级手段都用尽，对整个上下文进行全量压缩：

  • 使用 LLM 对全部历史生成一份综合摘要
  • 保留系统提示、关键工具定义、近期对话
  • 摘要作为新的"起始上下文"继续工作
  • 这是最后手段，代价最高但最彻底

设计精髓：Claude Code 的五段流水线体现了"能用规则不用模型，能用小模型不用大模型"的工程哲学。每一级压缩都比上一级更"贵"，但效果也更强。这种渐进式设计确保了成本和效果的平衡。

Codex CLI 采用"交接棒"隐喻：长对话就像接力赛，每一棒需要把足够的上下文传给下一棒，但不需要把每一棒的所有细节都带过去。

  • 近期保护：最近 N 轮用户消息保持原文不压缩
  • 远期摘要：更早的内容生成 handoff 摘要
  • 新上下文：摘要 + 近期原文，构成新的工作窗口
  • 隐含假设：越近期的内容越可能与当前任务相关

这种设计的优势是简单、可预测。劣势是 handoff 摘要的质量直接影响后续表现。

OpenCode 独特之处在于其可逆性设计：压缩不等于丢弃。

  • 时间戳标记：为每条消息打上时间戳，超过阈值的消息被"隐藏"而非删除
  • 结构化摘要：被隐藏的内容被转化为结构化摘要，保留关键决策和状态
  • 消息回放：压缩后回放最后一条用户消息，帮助模型重建当前任务焦点
  • 可展开恢复：需要时可以"展开"恢复被隐藏的原始内容

这种"先隐藏再摘要"的两步策略，在丢信息和省空间之间找到了一个新平衡点。

Cline 走的是"人机协同"路线，将压缩的控制权部分交还给用户。

  • /smol 手动触发：用户在觉得上下文冗余时主动触发压缩
  • Focus Chain 自动追踪：系统自动识别当前任务焦点，标记相关内容为"高优先级"
  • 选择性压缩：用户可以选择压缩哪些内容，保留哪些内容
  • 重要标记：用户可以标记某些上下文为"重要"，压缩时不会被触碰

双模式的好处是灵活，但也增加了用户的认知负担。

Cursor 的策略独特之处在于它不执着于"压缩"，而是用"搜索"来替代"压缩"。

  • 自动摘要 + 提示开新对话：当上下文过长时，主动建议用户开新对话
  • 历史可搜索：所有历史对话都被索引，可以通过搜索重新调取
  • Codebase 索引：代码仓库被索引为外部记忆，不必全部放入上下文
  • 动态发现：根据当前任务动态搜索相关的代码片段、文档和历史对话

这是一种"换一种思路"的方案：不压缩信息，而是让信息在需要时被找到。

Amp 采取了最激进的策略：不做压缩。

  • 核心信念：任何压缩都会丢信息，丢信息必然有风险
  • /handoff 命令：当上下文不够用时，开一个新线程继续
  • 单任务聚焦：每个线程只做一件事，做完就结束
  • 上下文隔离：不同线程之间天然隔离，不共享上下文

这种设计的前提是：用户能够合理地将大任务拆分为小任务。对于高度自主的 Agent，这可能不太现实。

MemGPT 将 LLM 上下文管理类比为操作系统的虚拟内存管理，这是一个深刻的范式转换。

  • 上下文 = RAM：当前上下文窗口就是"物理内存"，容量有限
  • 历史 = 磁盘：持久化存储的历史对话就是"磁盘"，容量无限
  • 换入/换出：Agent 自主决定何时将信息从"磁盘"加载到"内存"（page-in），何时将信息从"内存"卸载到"磁盘"（page-out）
  • 缺页中断：当 Agent 需要某个信息但不在上下文中时，触发"缺页中断"，从存储中检索
  • 自主管理：Agent 具有对自身记忆的完全管理权限，可以主动查询、存储、删除

这是目前最具理论深度的方案，但实现复杂度也最高。

实施陷阱：滑动窗口的代价

核心警告：看似简单的"滑动窗口截断 + stub 替换"策略，在 API 层面会引发灾难性的缓存失效问题。

滑动窗口策略的问题在于：每次截断最前面的消息后，整个消息序列的前缀都发生了变化。对于使用 prompt cache 的 API（Anthropic、OpenAI），这意味着每次请求都是一次全新的冷启动。

真实案例 // 一个真实任务的成本拆解
任务: "重构认证模块"
总成本: $77.30

// token 使用分布
cache_read: 12% // 缓存命中 — 正常
cache_write: 83% // 缓存写入 — 异常！几乎全在重建缓存
output: 5% // 实际输出

// 83% 的钱花在了"重新写缓存"上
// 每一步滑动窗口截断 → 前缀变化 → 缓存全失效
83%
cache_write 占比
滑动窗口导致每步缓存失效，
绝大多数 token 花费在重建缓存上

正确做法：压缩操作应该采用"追加式"而非"截断式"——新生成的摘要放在消息末尾，保持原有消息序列的前缀不变，从而最大化 prompt cache 的命中率。

5

七条共识原则

尽管各家策略不同，但从实践中提炼出了七条跨产品的共识原则。

1

分层压缩

不要一刀切。设计多级压缩策略，从轻量级的规则操作到重量级的 LLM 摘要，逐级加深。

2

成本递进

每一级压缩的计算成本应该递增。能用确定性操作（零成本）解决的，绝不动用 LLM。

3

增量摘要

压缩应该是增量式的——对新内容生成摘要，而非每次对全量内容重新压缩。追加而非替换。

4

真实 Token 计数

不要用"消息条数"估算上下文使用量。必须使用真实的 token 计数，因为一条工具输出可能包含数万 token。

5

用户特权

用户显式标记的内容应被视为"特权信息"，压缩时不可触碰。用户标记 > 系统判断。

6

保护近期

最近几轮对话是最可能与当前任务相关的。压缩应从最远的内容开始，保护近期上下文。

7

单调边界

压缩操作的"水位线"应该单调递增——已经压缩过的内容不应该被"展开回来"再压缩一次，避免信息振荡。

6

四级水位线方案

基于七条共识原则，提出一套标准化的四级水位线压缩方案。水位线单调递增，不可回退。

Tier 0 — 安全水位 (0-60%)

状态：上下文充裕，无需任何压缩操作。
行为：正常工作，所有消息保持原文。
监控：后台持续统计真实 token 使用量，为后续水位预警做准备。

Tier 1 — 预警水位 (60-75%)

状态：上下文开始紧张，需要主动管理。
行为：触发确定性裁剪 — 截断过长的工具输出、移除冗余返回值、压缩重复确认循环。
成本：零（纯本地操作，不调用 LLM）。
缓存影响：追加式操作，不破坏前缀缓存。

Tier 2 — 压缩水位 (75-90%)

状态：上下文紧张，需要 LLM 辅助压缩。
行为：对最早的对话段落生成结构化摘要，按任务边界折叠。摘要追加到消息末尾，保持前缀不变。
成本：中等（调用 LLM 生成摘要，但仅处理部分内容）。
缓存影响：追加式摘要，前缀缓存保持有效。

Tier 3 — 紧急水位 (90-100%)

状态：上下文即将耗尽。
行为：全量压缩 — 保留系统提示和工具定义，对全部对话历史生成综合摘要。建议用户开新对话。
成本：高（全量 LLM 摘要）。
缓存影响：全量重写，缓存失效，但这是最后手段。

7

云端三层架构

云端 Agent 的上下文管理需要在存储、工具和缓存三个层面系统性设计。

热存储

当前上下文窗口
全量原文
最高保真度
容量最有限

温存储

近期摘要 + 结构化索引
压缩后的关键信息
中等保真度
容量较充裕

冷存储

全量历史记录
可检索的原始对话
最低访问频率
容量无限

将信息按照访问频率和重要性分配到不同存储层：

  • 热存储：当前上下文窗口，包含全量原文，保真度最高但容量最有限
  • 温存储：压缩后的摘要和结构化索引，保真度中等，访问频率中等
  • 冷存储：全量历史记录，按需检索，容量几乎无限
  • 信息可以在层级间流动：热 → 温 → 冷（压缩降级），冷 → 温 → 热（检索升级）

不同类型的内容应该采用不同的压缩策略：

  • 工具输出：通常是最占空间的，优先裁剪。保留结构，删除冗余数据
  • 对话消息：按时间远近分层压缩，近期保留原文，远期生成摘要
  • 系统提示：通常不可压缩，但可以移除过时的指令
  • 代码片段：保留关键函数签名和逻辑，删除注释和空行

缓存策略是云端 Agent 成本控制的关键：

  • 前缀稳定性：所有压缩操作必须保持消息序列前缀不变
  • 追加式更新：新摘要追加到末尾，而非替换中间内容
  • 缓存预算：将 prompt cache 的命中率作为压缩策略的优化目标之一
  • 成本核算：cache_write 的价格是 cache_read 的数倍，必须监控
8

红线：不可触碰的底线

系统提示不可压缩。System Prompt 是 Agent 的"人格"和"规则"，任何对系统提示的压缩都可能导致 Agent 行为偏离预期，产生不可预测的后果。

完整的红线清单

  • ✖ 系统提示（System Prompt） — Agent 的核心指令集，压缩可能导致行为异常
  • ✖ 用户显式标记的重要信息 — 用户特权高于系统判断
  • ✖ 当前任务的活跃上下文 — 正在执行的任务所需的核心信息
  • ✖ 安全相关的约束条件 — 权限控制、数据访问规则等
  • ✖ 关键工具的定义和 Schema — 压缩工具定义会导致调用失败
9

哲学结论：Context Rot

Context Rot（上下文腐烂）

上下文不仅仅是信息的容器，更是模型的注意力资源。当上下文中充斥着大量低价值信息时，模型的注意力被稀释——这就是 Context Rot。

就像一间堆满杂物的房间让人无法专注工作一样，一个臃肿的上下文窗口让模型无法准确理解和执行任务。上下文压缩的本质不是"省空间"，而是"保护模型的注意力"。

注意力是稀缺资源

模型的注意力窗口有限。上下文越长、噪音越多，模型对关键信息的注意力就越弱。压缩的终极目标是信噪比最大化。

压缩即策展

好的上下文压缩不是简单的"删减"，而是一种"策展"——决定哪些信息值得模型花注意力去关注，哪些应该退到幕后。

记忆是 Agent 的灵魂

一个 Agent 的能力上限不取决于模型本身，而取决于它如何管理自己的记忆。优秀的上下文管理让弱模型胜过强模型。

工程 > 理论

上下文管理的每一条共识原则都来自工程实践的血泪教训，而非理论推演。在真实场景中，一个缓存失效就能让成本翻倍。

核心收获

分层渐进，能轻不重

上下文压缩应该像 Claude Code 的五段流水线一样，先用零成本的确定性操作，再用 LLM 辅助，最后才全量压缩。

追加式而非截断式

所有压缩操作都应该保持消息前缀不变，追加摘要到末尾，最大化 prompt cache 命中率。83% cache_write 的惨剧不要重演。

保护注意力，而非省空间

上下文压缩的本质是"策展"——让模型的注意力集中在最重要的信息上。Context Rot 比 Context Overflow 更可怕。

用户标记 > 系统判断

用户显式标记的内容是"特权信息"，压缩时绝不可触碰。系统提示、安全约束、工具定义同理——这些是红线。
