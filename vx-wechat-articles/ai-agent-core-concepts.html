全景：AI Agent 架构总览

▼
核心框架

什么是 AI Agent？

AI Agent 不是一个单一的模型，而是一个完整的系统。它由多个层次的组件协作组成：基础模型提供"思考能力"，工具提供"行动能力"，框架层提供"行为规则"，训练系统提供"进化能力"。理解 Agent，就是理解这些组件如何各司其职、协同工作。

一、Model：基础大模型

▼
定义

Model 是什么？

Model 即基础大模型（LLM），是 Agent 的"大脑"。它的核心能力是输入文本、输出文本——给定一段 prompt，返回一段补全。但 Model 本身有两个关键限制：无长时记忆（每次调用都是独立的）、无法自主循环（不会自己决定"再想一步"）。

类比理解

把 Model 想象成一个超级聪明但失忆的顾问。你每次找他咨询，他都能给出极高质量的回答，但他完全不记得上次你们聊了什么。而且，他不会主动联系你——你不找他，他就不工作。Agent 的其他组件，就是帮这个顾问"恢复记忆"和"主动工作"的系统。

Prompt 输入 → Model 推理 → 文本输出
▶ 为什么 Model 不能直接当 Agent？

因为 Model 缺少三个关键能力：（1）循环执行——它不会自己决定"结果不满意，再来一轮"；（2）外部感知——它无法读取文件、调用API、访问数据库；（3）状态管理——它不记得之前做了什么。这些能力全部由外层组件提供。

🏠

二、Scaffolding 与 Policy：框架与策略

▼
框架层

Scaffolding（脚手架）

Scaffolding 是 Agent 的"宪法"——它约束、定义 Agent 的整体运行规则。包括：Agent 可以使用哪些工具、在什么条件下允许执行什么操作、任务的边界在哪里、安全红线是什么。它是静态配置，在 Agent 启动前就已确定。

决策层

Policy（决策策略）

Policy 是 Agent 的"决策规则手册"。当 Agent 面对多个可选动作时，Policy 规定它如何挑选执行方案。例如：当工具调用失败时是重试还是换一个工具？当用户意图模糊时是追问还是猜测？Policy 可以是硬编码规则，也可以由模型动态生成。

任务输入 → Scaffolding 设定边界 → Policy 选择策略 → 执行动作

类比理解

Scaffolding 相当于公司的规章制度（不能做假账、不能泄露客户数据），Policy 相当于业务流程SOP（遇到客户投诉先道歉再调查再补偿）。前者是底线，后者是方法。

▶️

三、Harness：调度执行层

▼
核心组件

Harness 是 Agent 的"操作系统"

Harness 是整个 Agent 系统的调度中枢，负责三件核心事情：（1）调模型——组装 prompt、调用 Model、获取返回；（2）解析工具请求——当 Model 输出中包含工具调用指令时，Harness 负责解析参数、实际执行工具、将结果回填；（3）终止判定——判断任务是否完成、是否需要继续循环、是否超时或异常。

组装 Prompt → 调用 Model → 解析返回 → 执行工具 / 终止
关键循环

Agent Loop：Harness 的核心机制

Harness 驱动着 Agent 的核心循环——思考-行动-观察（Think-Act-Observe）。每一轮循环中：Model 生成思考和行动决策，Harness 执行行动并将结果作为观察反馈给 Model，如此反复直到任务完成。这个循环是 Agent 区别于单次 Model 调用的根本特征。

Harness 工程的复杂性

Harness 是 Agent 工程中最复杂的部分。它需要处理：并发工具调用、错误重试、超时控制、上下文窗口管理、token 预算控制、安全拦截等。一个优秀的 Harness，能让普通模型发挥出远超预期的 Agent 能力。

知识检查：Harness 的核心职责是什么？

训练大模型参数
提供 GPU 算力
调度 Model、解析工具请求、判定任务终止
存储用户数据

四、Context Engineering：上下文工程

▼
核心概念

不是 Prompt Engineering，是 Context Engineering

Context Engineering 是动态管控送入模型的 prompt 内容的工程实践。它不只是写一个好的 prompt（那是 Prompt Engineering），而是决定每一次调用时，模型应该看到什么信息：系统指令、对话历史、工具返回结果、检索到的文档、用户画像等。在上下文窗口有限的情况下，选择"放什么"和"不放什么"是核心挑战。

关键策略

上下文管理的三个维度

（1）选择——哪些信息需要进入上下文？（2）压缩——如何在有限窗口内塞入更多信息？（摘要、滑动窗口、关键信息提取）（3）排序——信息放在 prompt 的哪个位置？（模型对开头和结尾的信息更敏感）

用户输入 + 对话历史 + 工具结果 + 检索文档 → 最终 Prompt
▶ 为什么 Context Engineering 如此重要？

因为模型的输出质量完全取决于输入的质量。即使是最强的模型，给它错误的、过时的、不相关的信息，也会输出糟糕的结果。Context Engineering 决定了模型"看到的世界"，也就决定了它的"判断力"。很多 Agent 失败的案例，不是模型不行，而是上下文管理出了问题。

五、Tool Use 与 Skill：工具与技能

▼
执行能力

Tool Use（工具调用）

Tool Use 是 Agent 对接外部接口和软件的执行逻辑。工具可以是：搜索引擎、代码执行器、文件系统、API 接口、数据库查询等。Model 在推理过程中决定"我需要用什么工具"，Harness 负责实际执行工具调用并将结果返回给 Model。

复用单元

Skill（技能模块）

Skill 是封装好的、可复用的多步骤任务单元。与 Tool 不同，Skill 通常包含多个工具调用的编排逻辑、错误处理、条件分支等。例如"搜索并总结文档"是一个 Skill，它内部可能调用搜索工具、读取工具、总结工具。Skill 是 Agent 能力的模块化封装，可以跨任务复用。

单个原子操作 = Tool     多步骤编排 = Skill
Tool vs Skill 的关键区别

Tool 是原子操作（搜索、读文件、执行SQL），Skill 是复合任务（写一份市场分析报告、部署一个Web应用）。Tool 是 Skill 的构建块，Skill 是 Tool 的编排。一个成熟的 Agent 生态中，Tool 数量可能上百，但 Skill 数量通常更精炼。

知识检查：Skill 和 Tool 的本质区别是什么？

Skill 比 Tool 更快
Skill 是封装好的多步骤任务单元，Tool 是原子操作
Skill 只能由人类定义，Tool 由模型自动生成
Skill 需要联网，Tool 不需要

六、Sub-agent 与 Environment

▼
多智能体

Sub-agent（子智能体）

Sub-agent 是被主 Agent 调度的、独立承接细分任务的子系统。每个 Sub-agent 有自己的 Model、Tool 集合和 Policy。主 Agent 将复杂任务分解后分配给不同的 Sub-agent，各自独立执行后汇总结果。这是处理复杂任务的关键架构模式。

外部世界

Environment（交互环境）

Environment 是 Agent 感知和操作的外部场景。它包括：用户界面、文件系统、网络环境、其他软件系统等。Agent 通过 Tool 与 Environment 交互——读取环境状态、执行改变环境的操作。Environment 的反馈是 Agent 决策的重要输入。

主 Agent → Sub-agent A → Environment → 反馈 → 主 Agent 汇总
▶ 什么时候需要 Sub-agent？

当任务具有可并行性或需要不同专业能力时，Sub-agent 架构最有价值。例如："帮我做一份竞品分析"——一个 Sub-agent 负责搜索竞品信息，一个负责分析财务数据，一个负责生成报告。它们可以并行工作，各自使用不同的工具和策略。

知识检查：Sub-agent 架构的核心价值是什么？

将复杂任务分解，让独立子系统各司其职并行执行
减少模型的 token 消耗
让 Agent 能够联网
存储用户历史对话

七、Rollout、Reward 与 Trainer：进化闭环

▼
运行记录

Rollout（完整轨迹）

Rollout 是 Agent 单次任务的全流程运行记录。它记录了从任务开始到结束的每一个步骤：模型的每一次推理、每一次工具调用、每一次环境反馈、每一步决策。Rollout 是分析 Agent 行为、发现问题、优化策略的基础数据。

评估信号

Reward（奖励值）

Reward 是量化单次任务执行优劣的评分。它可以来自：人工评估（用户满意度）、自动指标（任务是否完成、耗时多少、token 消耗）、环境反馈（代码是否通过测试）。Reward 是 Agent 进化的方向信号——告诉系统"什么行为是好的"。

进化引擎

Trainer（训练器）

Trainer 依托 Rollout 轨迹和 Reward 评分，迭代优化模型参数。这就是 Agent 的"学习"过程：通过大量的任务执行（Rollout）和效果评估（Reward），不断改进模型的决策能力。典型的训练方法包括 RLHF、DPO 等。

任务执行 → Rollout 记录 → Reward 评估 → Trainer 优化 → 更强的 Model
进化闭环的意义

Rollout + Reward + Trainer 构成了 Agent 的自我进化闭环。这使得 Agent 不再是静态的系统——它会随着使用变得越来越强。这也是为什么数据飞轮（越多使用 → 越多数据 → 越好模型 → 越多使用）是 Agent 竞争的核心壁垒。

完整生命周期：12个概念如何协作

▼
全局视角

一次完整的 Agent 任务执行

当用户发出一个任务请求时，12个概念按以下顺序协作：Environment 接收请求 → Scaffolding 设定边界 → Policy 选择策略 → Context Engineering 组装上下文 → Harness 调度 Model 推理 → Model 生成决策 → Tool/Skill 执行动作 → Environment 返回反馈 → 循环直到完成 → Rollout 记录轨迹 → Reward 评估效果 → Trainer 优化模型。

🌎 Environment → 🏠 Scaffolding → ✅ Policy → 📦 Context → ▶️ Harness → 🧠 Model
🔧 Tool/Skill → 🌎 反馈 → 🔄 循环 → 📈 Rollout → ⭐ Reward → 📈 Trainer
▶ 概念之间的依赖关系

Model 是所有上层组件的基础——没有 Model，Harness 无法调度，Context Engineering 无处投喂。Harness 依赖 Context Engineering 来组装输入，依赖 Tool/Skill 来执行动作。Scaffolding 和 Policy 是 Harness 的配置参数。Sub-agent 是 Harness 的扩展——主 Harness 调度子 Harness。Rollout/Reward/Trainer 是整个系统的进化层，依赖前面所有组件产生的数据。
