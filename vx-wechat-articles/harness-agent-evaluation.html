一、开篇与核心问题

▼
核心背景

业务Agent协作链路的评测困境

某业务系统的内容生成链路由多个子Agent协作完成——图片理解、内容审核、文案生成、风格匹配等。这些Agent的prompt方案频繁迭代，每次变更后需要快速验证效果。核心矛盾在于：业务Agent迭代快（天级），但传统评测工程搭建慢（周级）。

传统评测痛点

六大难题

  • 启动成本高——搭建评测工程、写脚本、部署服务花一周
  • 人力密集——标注数据集、写分析脚本、出报告，每环节都需人工介入
  • 迭代慢——prompt改一行想看效果要等半天
  • 可复现性差——评测逻辑散落在各种脚本和Notebook里
  • 指标不统一——不同Agent各搞一套，无法横向对比
  • 工程化沉重——每换一个Agent就要新写一套评测代码
核心问题

如何用一种轻量、快速、可复用的方式，系统性地评测多个业务Agent？

业务Agent频繁迭代 → 需要快速验证效果 → 传统评测太慢太重 → 寻找轻量解法

二、核心解法：Harness工程搭建式评测

▼
核心定义

Harness 工程搭建式评测

用一个顶级Agent（Claude Code）作为Harness工程的搭建者和运行者，系统性地对业务Agent进行评测。评测Harness的本质是一套结构化的评估规则 + 执行流程。传统做法把它编码为Python脚本，而Harness式做法把它编码为Agent提示词——更灵活、更可读、更易迭代。

范式转变的核心洞察

评测Harness的本质是一套结构化的评估规则+执行流程。传统做法编码为Python脚本，Harness式做法编码为Agent提示词，由此获得5-10倍的效率提升和更高的分析质量。

传统做法 vs Harness式做法

对比维度 传统做法 Harness式做法
工作流 人写评测代码 → 跑脚本 → 看结果 → 改代码 → 再跑 顶级Agent搭建评测骨架，人只提供被测对象+做关键决策
启动周期 周级 天级
迭代速度 天级 小时级
核心编码 Python脚本 Agent提示词
关键优势

为什么Claude Code是合适的Harness搭建者？

  • 深度理解prompt——分析被测Agent逻辑，设计针对性评测维度
  • 代码生成——数据获取/处理脚本
  • 结构化输出——评测方案文档、评测Agent提示词、评测报告
  • 多轮协作——跨版本持续迭代保持上下文连贯
  • 数据分析——对跑批结果做统计归因对比

三、Harness工程整体架构

▼
三层架构

Harness 的骨架

整体由三层构成：规则层（评测方案）→ 数据层（黄金评测集）→ 执行逻辑层（评测Agent提示词），加上输出层（结果分析与报告）。

规则层：评测方案 → 数据层：黄金评测集 → 执行层：评测Agent提示词 → 输出层：结果分析与报告
搭建流程

Harness 搭建五步法

设计评测方案 → 构建评测集 → 编写评测Agent提示词 → 跑批执行 → 结果分析与报告

传统映射

与传统评测工程的类比

传统组件 Harness对应物
test_config.yaml评测方案.md（规则从配置文件变为自然语言文档）
test_data.json评测集Excel（数据格式统一，人可直接看懂）
test_runner.py（数百行）评测Agent提示词（数千字，执行逻辑从代码变为Prompt）
conftest.py + fixturesGT标注 + ground_truth字段（预期结果内嵌在数据中）
report_generator.pyCC实时分析（报告生成从脚本变为交互）
requirements.txt + CI评测平台一键跑批（零部署成本）
职责分工

三方协作模型

  • 人：负责GT标注、方案审核、最终决策（不写评测脚本，不手动计算指标）
  • Claude Code：负责Harness全链路搭建 + 结果分析（不做批量推理主循环）
  • 评测平台：负责批量执行引擎逐行调用（不做方案设计和指标汇总）

四、统一评测指标框架

▼
指标体系

三层指标体系（L1 / L2 / L3）

为实现不同Agent之间的标准化横向对比，设计了三层递进的指标体系：

L1

通用基础指标（所有Agent必报）

  • 输出格式合规率——JSON可成功解析的比例
  • 字段完整率——必要字段均存在的比例
L2

按能力类型选用

  • 分类判断 → 分类准确率
  • 二元决策 → 召回率/精确率
  • 数值提取 → 精确匹配率
  • 连续评分 → MAE + 分档一致率
  • 文本生成 → LLM-as-Judge 1-5分
L3

Agent专属指标

每个Agent可在L1+L2基础上追加专有指标。例如：文案生成Agent的违禁词清洁率、关键信息保留率。

确定Agent能力类型 → 从L2菜单勾选指标 → 追加L3专属指标 → 设定目标阈值
新Agent接入指标选型流程

新Agent接入时，按"确定能力类型 → 勾选L2指标 → 追加L3指标 → 设定阈值"四步完成指标配置，实现即插即用的标准化评测。

知识检查 2：三层指标体系中，L2指标的核心特征是什么？

所有Agent必须强制使用
由评测平台自动生成
按Agent的能力类型（分类、二元决策、文本生成等）选用
仅用于最终报告输出

五、Harness各层的搭建方法——四角色模型

▼

Claude Code在Harness搭建过程中扮演四个角色，分别对应四层架构的搭建任务。

角色一：方案架构师 规则层

评测方案设计

输入：被测Agent的prompt文件 + 业务上下文描述

CC输出：完整的评测方案文档（含维度、指标、阈值、数据集要求、错误分类体系）+ 边界用例建议（CC分析prompt逻辑后主动提出应覆盖的场景）

实际效果：从一个prompt文件到一份完整评测方案，大约10分钟的交互。

角色二：数据工程师 数据层

黄金评测集构建

CC在数据层承担四件事：

  • 数据获取——编写脚本调用业务接口批量拉取候选数据
  • 数据处理——格式化为评测所需的JSON结构
  • GT辅助标注——对分类型指标CC先给建议标注，人工复核
  • 评测集打包——生成评测平台可直接消费的Excel（含system.question列）
关键设计：system.question列

每行数据包含被测Agent所需的全部输入字段 + ground_truth（人工标注的黄金答案）。评测Agent读取这一列即可获得输入和预期输出，无需额外配置。

角色三：Harness工程师 执行逻辑层

评测Agent提示词编写（最核心的创新）

核心思路：把传统的评测脚本（Python/Java）替换为一份评测Agent提示词。评测逻辑从"代码"变为"自然语言指令"，一个Agent来评测另一个Agent。

读取system.question → 调用被测Agent → 解析输出 → 硬规则检查 → LLM打分 → 输出JSON
提示词结构模板

评测Agent提示词的五段式结构

角色定义 → 工具声明 → 约束 → 工作流程 → 输出Schema

设计优势

四大优势

  • 逻辑可读——评测逻辑以自然语言写在提示词里，无需读代码
  • 快速迭代——发现评测逻辑有误，改一段文字就行，不用改代码重部署
  • 统一执行——所有Agent的评测逻辑结构一致，只改内容不改框架
  • 评测即文档——提示词本身就是评测标准的完整说明
角色四：数据分析师 输出层

结果分析与报告

CC在分析中的增值体现在三方面：

  • 自动识别pattern——不只报数字还归因。例如："18条误过滤中12条都是把某评分维度<60当过滤条件"
  • 跨批次对比——和上一版结果对比，明确哪些指标进步/退步
  • 给出可操作建议——不只是"分数低"，而是"建议在prompt第三段加入明确的过滤条件边界"
注意：角色三"评测Agent"是整个方案最核心的创新

用Agent评Agent，评测逻辑从Python脚本变为提示词——这不仅是工具的变化，更是思维范式的升级。评测即Prompt，Prompt即文档。

六、关键实践经验

▼
评测集设计

四大原则

  • 小而精——20-55条足够覆盖所有边界场景；200+条但都是简单case反而是反例
  • 分布均衡——正/负例比例合理，边界场景必须有
  • GT可复核——每条GT标注有据可查
  • 版本化管理——评测集跟随被测prompt版本变更
常见Bug

评测系统四大常见问题及修复

Bug 1：匹配逻辑过严

语义等价的判定原因被判错

修复：GT ⊆ AI超集匹配

Bug 2：硬编码规则误报

排除列表不全导致误判

修复：改为动态语义比对

Bug 3：Token截断

输出超长被评测平台截断

修复：正则容错提取关键字段

Bug 4：GT覆盖缺口

新增选项未在GT中体现

修复：更新GT标注
迭代节奏

三阶段迭代策略

  • v1：基本逻辑跑通（调试模式带推导过程）
  • v2：切换为跑批模式（纯JSON输出），修复首批评测逻辑bug
  • v3+：基于实际结果持续调优
LLM-as-Judge

文本生成类Agent的评审方法

对文本生成类Agent无法精确匹配GT时，用LLM做评委。在评测Agent提示词中嵌入评分标准（1-5分rubric），评测Agent同时扮演"执行者"和"评委"。

有效的rubric设计要求：每个分值必须有具体可区分的判定标准，避免"好/较好/一般"这类主观描述，分值之间的差异应该一个正常人也能判断。

实际踩坑

"评测Agent调被测Agent"的四大坑

  • 评测Agent忘记调用工具——在Constraints中强调"必须先调用工具"
  • 工具参数传递失败——在提示词中显式写明参数构造逻辑
  • 评测Agent重试耗尽token——添加"禁止重试"约束
  • 输出截断——减少推导过程，只输出最终JSON

知识检查 3：评测Agent提示词迭代中，"匹配逻辑过严"的正确修复方式是什么？

增加更多硬编码排除规则
直接跳过匹配失败的case
人工逐条审核所有结果
采用GT ⊆ AI超集匹配，允许语义等价的变体

知识检查 4：关于评测集设计，以下哪项是正确的？

评测集越大越好，至少200条以上
20-55条小而精即可，关键是覆盖所有边界场景
只用正例即可，负例不重要
评测集一旦完成就无需更新

质量保障

四个维度的保障

  • 覆盖性——CC不会遗漏任何数据行
  • 一致性——同样的评测标准，CC不会因疲劳而评分漂移

八、适用场景与局限

▼
适用场景

四大场景及推荐程度

  • Prompt迭代验证 — ★★★★★ — 改prompt → 跑批 → 看报告，闭环最快
  • 多Agent横向对比 — ★★★★★ — 统一指标框架 + 相同评测流程
  • 新Agent上线前验收 — ★★★★ — 系统性覆盖，不依赖人工抽检
  • 线上问题复盘 — ★★★ — 可快速构造问题用例验证
局限性

四个需要关注的局限

  • LLM-as-Judge本身有偏差——对关键决策用人工抽检兜底
  • 评测集规模受限于人工GT——小而精优于大而糙，20-55条覆盖边界即可
  • 依赖评测平台稳定性——token截断、API超时需做容错
  • 首次搭建有学习成本——第二个Agent起复用率很高
重要提醒

LLM-as-Judge并非万能。对于关键决策场景（如内容安全审核），务必保留人工抽检环节作为兜底机制。

各步骤详情

五步法详解

  • Step 1 准备工作（5分钟）：准备prompt文件 + 确认接口信息
  • Step 2 设计评测方案（10-30分钟）：CC输出维度、指标、阈值、数据集要求、错误分类
