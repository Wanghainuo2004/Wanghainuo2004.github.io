拆解大模型核心操作的数学与 Infra 优化逻辑

概述

本文将拆解大模型中几个核心操作（RMSNorm、Softmax、Causal Mask、Sampling）背后的数学与 Infra 优化逻辑。

💡 核心观点

Infra 优化，本质上就是在用数学上的等价变换，或者对精度的适度妥协，去换取更高的硬件利用率和极致的推理速度。

RMSNorm

均方根归一化

解决深层网络数值不稳定问题，防止梯度消失/爆炸和硬件溢出。

Softmax

概率映射函数

将 logits 转换为概率分布，是 Attention 机制的核心。

Causal Mask

因果掩码

确保自回归模型只能看到当前及之前的 token，防止信息泄漏。

Sampling

采样策略

从概率分布中选择下一个 token，包括 greedy、top-k、top-p 等方法。
📐

RMSNorm - 均方根归一化

大语言模型通常包含数十甚至上百个堆叠的隐藏层。输入张量在经过连续的矩阵乘法和加法操作后，其数值的分布范围会发生剧烈变化。

为什么需要归一化？

算法收敛困难

数值过大或过小影响训练稳定性

极端数值会落入激活函数饱和区导致梯度消失，或不受控放大引发梯度爆炸。

硬件溢出风险

低精度浮点格式的限制

FP16 数值上限仅 65504，BF16 尾数仅 7 bit，都可能导致 NaN 或截断误差。

从方差到标准差

要理解 RMSNorm，先要理解方差和标准差。

方差的诞生：为什么用平方而不是绝对值？

问题：正负波动会互相抵消，直接相加得到 0。

绝对值方案：符合直觉，但绝对值在零点不可导，无法用于微积分优化。

平方方案：既能消除负号，又是平滑的抛物线，完美支持求导。

Var(X) = Σ(xᵢ - μ)² / N
标准差：还原真实单位

方差的问题：单位被平方了（气温波动 40°C，方差却是 1600 平方摄氏度）。

解决方案：对方差开根号，还原为原始单位。

σ = √(Var(X)) = √(Σ(xᵢ - μ)² / N)

几何意义：标准差本质上就是高维空间里的直线距离（勾股定理的推广）。

Z-score 标准化

数组每个元素先减均值再除以标准差后，新数组的方差变为 1，均值变为 0。

z = (x - μ) / σ
🔑 关键性质

方差的常数乘法性质：如果每个元素乘以常数 k，新方差 = 原方差 × k²

RMSNorm 的工程实现

RMSNorm 是 LayerNorm 的简化版本，省略了减均值的步骤，只做缩放：

RMSNorm(x) = x / √(mean(x²) + ε) × γ

优势：计算量更少，效果相当，已成为现代大模型的标准选择。

🎯

Softmax

Softmax 是 Attention 机制的核心，将 logits 转换为概率分布。

softmax(xᵢ) = exp(xᵢ) / Σexp(xⱼ)
工程实现的挑战

数值稳定性：直接计算 exp(x) 可能溢出。解决方案：减去最大值。

softmax(xᵢ) = exp(xᵢ - max(x)) / Σexp(xⱼ - max(x))

性能瓶颈：exp 是超越函数，需要 SFU 单元计算，吞吐量仅为普通 ALU 的 1/4。

🎭

Causal Mask

在自回归生成中，模型只能看到当前及之前的 token，不能"偷看"未来。

实现原理

通过将未来位置的 attention score 设为 -∞，经过 softmax 后这些位置的概率变为 0。

工程优化：在 kernel 层面直接跳过这些位置的计算，而不是先算再 mask。

🎲

Sampling 采样策略

从 softmax 输出的概率分布中选择下一个 token。

策略 方法 特点
Greedy 选概率最高的 确定性，但可能重复
Top-K 从概率最高的 K 个中采样 平衡质量与多样性
Top-P 从累积概率达到 P 的集合中采样 自适应候选集大小
Temperature 调整 logits 的缩放 控制分布的尖锐程度
⚙️

硬件视角：SFU 与 FlashAttention-4

SFU（Special Function Unit）

GPU 中专门处理超越函数（exp、log、sin、cos 等）的硬件单元。

⚠️ 瓶颈所在

每个 SM 有 128 个常规 FP32 单元，却只有 16 个 SFU。SFU 吞吐量仅为 ALU 的 1/4。

RoPE 的 cos_sin_cache 优化

问题：sin/cos 是超越函数，SFU 吞吐量低。

方案：初始化时预计算好 cos_sin_cache，运行时直接查表。

本质：用几 MB 的内存换掉每次 forward 几百万次的 sin/cos 调用——空间换时间。

FlashAttention-4：软件模拟 exp

当 SFU 成为瓶颈时，FA4 用通用 ALU 分担部分 exp 计算。

💡 软硬协同分流

75%~90% 的指令调用原生 SFU，溢出的 10%~25% 用 ALU 通过多项式逼近模拟。

为什么不 100% 用 ALU 软算？

软件模拟会消耗宝贵的寄存器，全部软算会导致寄存器溢出（Spill）从而反噬性能。

FA4 通过极其精准的比例调优，让 SFU 和 ALU 首次在超越函数的计算上实现了完美的并行运转。

4
